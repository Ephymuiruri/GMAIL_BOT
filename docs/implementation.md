# Implementation notes — design trade-offs

This document captures the architectural decisions made while designing the
MCP-based email triage bot, and the reasoning behind each one. `CORE_BUILD.md`
describes an earlier, non-MCP version of this project (plain function calls,
no LLM tool-use loop) and is now superseded by this design for anything
architecture-related. It's still useful as reference for raw Gmail API
mechanics (OAuth flow, label creation, etc.).

---

## Architecture

- **`gmail_client.py`** is the MCP *client* and orchestrator. It:
  - Authenticates with Gmail and fetches emails
  - Talks to the LLM (Gemini), sending tools + preferences + email batch
  - Drives the tool-call loop — when the LLM calls a tool, the client relays
    that call to the MCP server and returns the result to the LLM
- **`gmail-mcp/gmail.py`** is the MCP *server*. It exposes tools (`label`,
  `archive`, `star`, or a combined `process_email`) that actually execute
  Gmail API calls. The server does not decide anything — it only acts on
  what the LLM (via the client) tells it to do.

This mirrors true MCP roles: the server exposes capabilities, the client
orchestrates the conversation between the LLM and the server.

---

## Cost model: what actually costs tokens

- Tokens are billed for text sent to the model (input) and text generated by
  the model (output). Network/API latency (e.g. waiting on Gmail's servers)
  costs **time**, not tokens — no LLM involvement in that part of the round
  trip.
- A tool-call round trip costs tokens twice: once when the LLM **generates**
  the call (output), and once when the **result** is fed back in as input on
  the next turn. Verbose tool results are a hidden, recurring cost since they
  re-enter context on every subsequent turn of the conversation.

---

## Prompt structure and caching

- LLM calls are stateless — "memory" across calls is just re-sending state
  (tools, preferences, labels) yourself. There is no need for a vendor
  "memory" feature for this use case.
- Prompt/context caching (Gemini context caching, Anthropic prompt caching)
  doesn't reduce the number of input tokens sent — it reduces **cost and
  latency** for a prefix that's byte-for-byte identical across calls.
- Caching requires the stable content first, volatile content last. Order
  chosen, from least to most volatile:

  **tools → instructions → labels → email**

  Labels are more stable than instructions might suggest, but not stable
  enough to be trusted as immutable — they can change if the user edits
  their preferences, so they still sit after instructions/tools in the
  prefix, ahead of only the email itself.
- Labels are passed to the LLM as plain text in the system prompt (not
  fetched via a separate "list labels" tool call). This saves a full
  round-trip per conversation and constrains the LLM to valid labels
  directly in the context it's reasoning over.

## Batch sizing

- Fixed batch size (whether 1 or N) is the wrong mental model. Not every
  email needs equal "thinking effort."
- **Pre-filter/triage** emails cheaply before the LLM call, using sender
  reputation and keyword rules — many emails (github notifications, receipts,
  newsletters) can be classified with zero AI cost.
- Emails that pass the pre-filter go to the LLM in **dynamic batches sized
  by assumed importance**:
  - High-importance / ambiguous emails → small batches (or solo), to reduce
    context bleed between unrelated emails and keep hallucination risk low,
    and to make failures easy to isolate/retry.
  - Low-importance emails → larger batches, since a misclassification is
    low-stakes.
- Anything auto-labeled by the pre-filter without LLM review is flagged
  distinctly (e.g. `source=rule` in the decision log) so misfires are
  auditable rather than silent.

## Tool design

- `label(email_id, label_name)` needs both parameters — the server has no
  way to know *which* email to act on otherwise.
- Considered but rejected: renaming tool `name` fields to short enums
  (`"1"`, `"2"`, `"3"`) instead of descriptive names (`label`, `archive`,
  `star`).
  - Tool `name` is sent once as part of the (cacheable, stable) tool list,
    and appears exactly once per call as a single JSON field in the LLM's
    output — the savings from shortening it are a rounding error compared
    to other optimizations below.
  - Not worth the loss of clarity for negligible token savings.
- Considered but rejected: a separate "list labels" tool call — costs an
  extra round trip for information that's cheap to just inline in the
  system prompt (see above).
- **Adopted: combine `label`/`archive`/`star` into a single
  `process_email(email_id, label=None, star=False, archive=False)` tool.**
  One tool call per email instead of up to three — cuts call+result
  overhead roughly 3x per email as inbox size grows, which is a much larger
  win than any tool-name-shortening scheme.
- **Adopted: short/local email identifiers in the prompt.** Gmail's real
  message IDs are long opaque strings that would otherwise be repeated by
  the LLM in every tool call for that email. Instead, assign a short local
  ID (e.g. `1`, `2`, `3`, or an index) when building the batch prompt, and
  maintain a local mapping back to the real Gmail ID on the client side
  before making the actual Gmail API call.
- Tool call **results** should be terse (e.g. `{"status": "ok"}`) rather
  than echoing back full email content, since every result re-enters
  context and gets billed as input tokens on all subsequent turns.

---

## Summary ranking of optimizations (largest to smallest impact)

1. Pre-filtering / sender rules — skips the LLM entirely for many emails
2. Combined `process_email` tool — cuts round trips ~3x per email
3. Short local email IDs — avoids repeating long opaque IDs per call
4. Prompt/prefix caching with correct ordering — cheaper repeated prefix
5. Labels inlined in system prompt vs. a separate tool call — saves one
   round trip per conversation
6. Terse tool-call results — avoids re-billing verbose text every turn
7. Shortening tool `name` fields to enums — negligible, not adopted
